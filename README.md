# 🌍 漫遊異世界 Wanderworld

> 同一份遊戲規格，三個 GPT-5.6 模型，一次生成。誰最快？誰最省？誰交付得最完整？

這個 demo 比較 GPT-5.6 Sol Low、Luna Max 與 Terra XHigh 使用同一份 prompt、同一批 `assets/`，以 one-shot 方式製作《漫遊異世界》的成果。目標是一款可用滑鼠及手機觸控遊玩的低多邊形 3D 散步遊戲：玩家穿梭四個區域、觀察動物、與路人對話，並把發現記錄在旅人手札。

## ✨ V2 結果一眼看懂

| | ⚡ **Sol Low** | 🏆 **Luna Max** | 🧭 **Terra XHigh** |
|---|---:|---:|---:|
| 完成時間 | **11m 22s** | 27m 42s | 24m 27s |
| API 等值估算成本 | US$3.22 | **US$1.63** | US$3.63 |
| 總 tokens | **3.07M** | 7.20M | 7.59M |
| Output tokens | **24.1K** | 85.3K | 57.6K |
| `dist/` 大小 | **6.4 MB** | 8.7 MB | 7.8 MB |
| 實作程式規模¹ | **107 行** | 1,462 行 | 928 行 |
| 定位 | 極速、精簡 | 完整、可稽核 | 平衡型 |

> ¹ 以各成品 `dist/assets/app.js` 的實體行數計算；壓縮程度不同，因此只反映交付形式，不等同程式品質。成本是依任務當時的標準 API 價格與本機 session log 推算，實際 Codex 訂閱計費可能不同。

### 🥇 三個最鮮明的結論

- **追求速度：選 Sol Low。** 11 分 22 秒完成，比 Luna 快約 59%，亦比 Terra 快約 53%；成品體積最小。
- **追求成本與交付透明度：Luna Max 最突出。** 成本最低，並在統計中明確列出 4 個區域、14 位 NPC、14 種動物、13 個地標／事件及 98 個 GLB 模型。
- **追求中間路線：Terra XHigh。** 完成時間與程式規模介乎兩者之間，但本次 API 等值成本為三者最高。

三個版本均包含純靜態輸出、本地 Three.js dependency、模型與授權檔案，並實作點擊／觸控移動、角色互動、旅人手札及 `localStorage` 探索紀錄。以上比較聚焦可由 repo 核對的產物與 session 數據，不把程式行數或 token 數直接視作品質分數。

## 🧪 這個 demo 的生成流程

### 第一次 prompt（GPT-5.6 Sol Low）

```text
based on all the 素材 from assets, create a prompt to create a game.
  the game should be like:
  - 一個人到處走
  - 到處走的時候希望看看不同的場景，不同的動物，不同的人，不同的事物等等
  - 然後會跟一些路人對話
  要用mouse, mobile friendly
```

## 技術與交付形式

- 製作成純靜態網站。
- 使用 HTML、CSS、JavaScript 與 Three.js。
- 不需要後端、登入、資料庫或 API。
- 不依賴 Node.js 才能執行成品。
- 所有 runtime dependency 必須放在網站目錄內，避免依賴 CDN。
- 最終輸出至 `dist/`。
- `dist/index.html` 應可透過任何靜態 HTTP server 開啟。
- 可直接部署到 Cloudflare Pages、GitHub Pages 或 Netlify。
- 不要修改原始 `assets/`；把實際用到的素材複製到 `dist/assets/`。
- GLB 模型旁的 `Textures` 目錄必須一併保留，維持正確相對路徑。
- 加入簡短 README，說明操作方式、本機預覽與部署方法。

- 郊區住宅與道路
- 奇幻城鎮建築、攤檔、噴泉、風車與水車
- 森林、花草、蘑菇、岩石、樹木及雪地
- 墓園、墓碑、地窖、蠟燭、南瓜與幽靈

不需要同時把 583 個模型全部載入場景。應從每個主要素材包挑選有代表性的模型，讓各類素材都有合理用途，同時控制效能。

不要把不同素材隨機堆在一起。每個模型都應服務於場景敘事、地標辨識、角色活動或互動事件。

## 世界設計

建立一張連續、可以步行穿越的小型開放世界。玩家不需要經過 loading screen，便能在四個有明顯視覺差異的區域之間移動。

### 1. 晨光郊區

使用：

- city-suburban 建築
- 住宅小徑、driveway、圍欄和花盆
- 一般汽車、的士、貨車、救護車、消防車或警車
- blocky people
- 貓、狗、蜜蜂、小鳥等動物

小事件：

- 郵差正在找正確地址
- 小孩在找走失的小狗
- 的士司機正在等待乘客
- 消防員在檢查街邊設施
- 一隻貓躲在住宅旁

### 2. 奇幻市集

使用：

- fantasy-town 建築零件
- 市集攤檔、推車、旗幟、燈籠
- 噴泉、風車與水車
- fantasy-town 道路、樹木、岩石和圍欄
- blocky people 與 platformer 奇幻角色

小事件：

- 商人正在展示一顆奇怪寶石
- 麵包師傅忘記風車的鑰匙放在哪裡
- 旅人圍在噴泉旁交換故事
- 一位角色請玩家替他觀察水車是否仍在運作

### 3. 林間高地

使用：

- platformer 草地 terrain
- 樹木、花、蘑菇、草叢、岩石
- 小路、橋狀地形、梯子、門、按鈕及拉桿
- Cube Pets 動物
- 金幣、星星或寶石作為可發現物

這裡是動物最集中的區域。讓鹿、狐狸、兔子、熊貓、河狸、蜜蜂、企鵝等出現在適合牠們的小型棲息地。

動物應有簡單 idle 行為，例如：

- 原地張望
- 在小範圍慢慢走動
- 靠近食物或同類
- 玩家接近時望向玩家
- 部分膽小動物稍微走開

### 4. 暮色墓園

使用：

- graveyard 建築、圍牆、墓碑與地窖
- 枯樹、松樹、岩石、南瓜、乾草和長椅
- 蠟燭、燈柱、火盆與燈籠
- keeper、ghost、skeleton、vampire、zombie

氣氛可以神秘，但不要恐怖或血腥。角色應帶有幽默感。

小事件：

- 守墓人正在找遺失的鏟子
- 骷髏忘記自己原本坐在哪張長椅
- 幽靈想知道市集今天是否熱鬧
- 吸血鬼抱怨墓園的燈太亮
- 一隻動物在墓碑附近睡覺

## 玩家與鏡頭

- 使用第三人稱斜上方視角。
- 選擇一個 blocky character 作為玩家角色。
- 玩家走路時角色面向移動方向。
- 鏡頭平滑跟隨，不要突然跳動。
- 地形、建築、圍欄及主要物件需要基本碰撞。
- 防止玩家穿牆、跌出地圖或被場景卡死。
- 如果模型沒有走路動畫，可以用輕微上下擺動及身體傾斜表現步行。

## 滑鼠操作

桌面版必須能完全只用滑鼠遊玩，不可強迫使用鍵盤。

- 點擊地面：角色自動走向指定位置
- 按住並拖曳：旋轉鏡頭
- 滾輪：拉近或拉遠鏡頭
- 點擊路人：自動走近，進入範圍後開始對話
- 點擊動物：自動走近並顯示觀察資訊
- 點擊互動物件：走近後觸發事件
- 點擊對話選項：選擇回應
- 點擊畫面空白處或關閉按鈕：關閉對話／觀察卡

可額外支援 WASD、方向鍵及 E 鍵，但這只能是輔助操作。

點擊移動時，在地面短暫顯示清楚但不突兀的目的地標記。

## 手機與觸控操作

遊戲必須 mobile friendly，支援手機直向和橫向畫面。

- 單點地面：移動
- 單點人物、動物或物件：移動並互動
- 單指拖曳空白位置：旋轉鏡頭
- 雙指縮放：調整鏡頭距離
- 不要依賴 hover
- 所有按鈕的可點擊範圍至少 44×44 CSS pixels
- 對話選項不可太小或貼近螢幕邊緣
- 使用 `touch-action` 與 Pointer Events 正確處理滑動和點擊
- 避免拖動鏡頭後誤觸地面移動
- UI 必須避開手機 safe-area
- 禁止頁面在遊戲操作時意外滾動或縮放
- 根據裝置效能降低 pixel ratio、陰影和同時顯示的模型數量

不要加入固定虛擬搖桿，除非點擊移動在實際測試中無法提供良好體驗。

## 對話系統

至少加入 10 位可對話 NPC，分布於四個區域。

每位 NPC 應具備：

- 名字
- 身分或職業
- 2 至 4 段簡短對話
- 至少一個玩家回應選項
- 與所在場景相關的內容
- 對附近角色、動物、地標或事件的提及

對話使用自然、帶一點幽默感的繁體中文。

對話 UI：

- 顯示 NPC 名字
- 顯示清楚易讀的繁體中文內容
- 每段內容保持簡短
- 提供 1 至 3 個大型回應按鈕
- 支援滑鼠和觸控
- 對話時暫停玩家移動
- 鏡頭可輕微移向 NPC，但不要突然大幅轉動

範例：

守墓人阿柏：
「你有看到一把鏟子嗎？它本來靠在長椅旁。至少我記得那是一把鏟子，不是一個非常安靜的骷髏。」

玩家選項：

- 「我替你留意一下。」
- 「那邊的骷髏好像拿著甚麼。」
- 「我只是路過看看風景。」

選擇可以改變下一句對話，但不需要製作龐大的分支劇情。

## 探索與發現

加入一本簡單的「旅人手札」，記錄玩家已經發現的內容：

- 到訪過的區域
- 看過的動物
- 對話過的人物
- 發現過的地標

第一次發現時顯示短暫通知，例如：

- 「新發現：河狸」
- 「新地點：月影墓園」
- 「新朋友：市集商人米拉」

進度儲存在 `localStorage`，重新整理後仍然保留。

不要製作複雜任務系統。可以有輕量的小請求和環境事件，但探索本身就是主要獎勵。

## UI 與視覺方向

- 整體使用溫暖、童話感、低多邊形風格。
- UI 應簡潔，不要遮蓋大部分 3D 畫面。
- 首次進入時顯示一張簡短操作提示卡。
- 左上或右上顯示目前區域名稱。
- 提供手札按鈕、音效開關及重新開始選項。
- 桌面與手機使用同一套響應式介面。
- 使用系統字體或把字體檔放在本地，不能依賴 Google Fonts。
- 加入 loading screen，顯示素材載入進度。
- 若個別模型載入失敗，不應令整個遊戲停止。

## 效能要求

- 只載入遊戲實際使用的模型。
- 重複物件盡量 clone 或使用 instancing。
- 合理限制陰影數量與距離。
- 遠處小物件可隱藏或降低更新頻率。
- 動物與 NPC AI 只在玩家附近更新。
- 桌面及主流手機瀏覽器均應保持可遊玩。
- 正確處理視窗 resize、裝置旋轉及 WebGL context。
- 避免每幀建立新物件或執行昂貴的 raycast。
- 點擊判定只檢查地面與可互動物件。

## 聲音

如果 `assets/` 沒有音訊素材，不要從網路下載來歷不明的音樂。

可以：

- 完全不加入聲音；或
- 使用 Web Audio API 生成非常簡單的 UI 提示音與環境音

瀏覽器首次互動前不可自動播放聲音。

## 明確不要做的事情

- 不要建立後端
- 不要要求 npm install 才能執行最終成品
- 不要依賴外部 CDN
- 不要加入廣告、登入或付費功能
- 不要把遊戲做成射擊遊戲
- 不要加入血腥或驚嚇內容
- 不要一次載入全部 583 個 GLB
- 不要用大量文字說明取代真正可玩的探索
- 不要只支援鍵盤
- 不要使用手機 hover 互動
- 不要生成新的 3D 模型取代現有素材
